# NexAgent

OpenClaw 出来之后，很多人都会自然地问一个问题：既然已经有“小龙虾”了，为什么还要再做一个 `NexAgent`？

这个问题表面上是在问“有没有必要重复造轮子”，但我觉得它真正问的是另一件事：当 agent 开始真正进入工作和生活之后，我们到底想要一个什么样的系统？

在“养龙虾”的过程中，很多人第一次很强烈地感受到，agent 已经不是过去那种只会对话的东西了。它会写代码，会串流程，会调用工具，有时候甚至会长出一些你事先没有明确设计过的能力。这个阶段很容易让人兴奋，因为你会突然发现，软件的形态好像在发生变化。以前我们做软件，往往是先把功能定义好，再把代码写好，再把产品交付出去；现在你会隐约看到另一种东西正在出现，用户说一句话，系统开始现场理解、现场组合能力、现场产出结果。

但我觉得，真正有意思的部分，其实不是这一层“它很神奇”。真正有意思的，是下一层问题。

如果这个 agent 不只是一个实验性的玩具，而是一个长期存在的系统，它该怎么活着？如果它不是只在一个终端里偶尔跑一轮，而是要进入 Telegram、Feishu、Discord 这些真实入口，它该怎么组织自己的会话？如果它会长期积累上下文、经验和方法，这些东西该放在哪里？如果它未来还会继续长出新的能力，我们又该怎么给这种变化一个稳定的结构？

我觉得做到这里，问题就已经变了。它不再是“怎么做一个 agent demo”，而是“怎么做一个长期运行的 agent 系统”。

`NexAgent` 就是在这个问题上开始的。

它的出发点，不是想重复做一个 OpenClaw，也不是想证明“我们也能做一个差不多的东西”。恰恰相反，我们真正想验证的是：在“养龙虾”这件事已经把问题打开之后，我们能不能把其中一些还比较模糊的直觉，继续往系统设计的方向推进一步。

比如说，大家在“养龙虾”的时候都会很快遇到几个问题。第一，记忆到底是什么？很多系统会说自己有 memory，但实际上的 memory 往往只是一些标签、一些事件，或者一些松散的摘要。可是真正有价值的东西，常常不是“你记得我是谁”，而是“你记得我是怎么做事的”。第二，agent 的主动性该怎么来？我们很容易把“主动性”想得很玄，觉得要有复杂的规划、复杂的反思、复杂的自治，但有时候事情没那么玄，可能一个合适的调度结构，一个定时器，一个后台执行机制，就已经足够让系统表现出完全不同的形态。第三，agent 学到的新东西，该沉淀在哪里？是不是所有东西最后都只能变成一段 prompt，或者一条 memory？如果是这样，系统很快就会糊成一团。

这些问题叠在一起，会把你逼到一个地方：你不得不认真思考这个 agent 的内部结构。

这也是为什么我一直觉得，自己做一个 agent 仍然是很有价值的。不是因为别人没有做，也不是因为别人做得不够好，而是因为很多理解，如果你没有亲手踩过那些坑，其实很难真正长出来。你当然可以看别人的系统、看别人的代码、看别人的分享，也一定能学到很多东西，但“知道”跟“做过”之间，还是有一段距离。只有自己真的做过一个长期运行的 agent，你才会知道哪些问题属于 prompt，哪些问题属于 runtime，哪些问题属于工具层，哪些问题最终只能进入代码层。

所以在我看来，`NexAgent` 更像是一个继续追问的结果。OpenClaw 已经把门推开了，`NexAgent` 关心的是推开门之后，里面的房间到底该怎么搭。

## 为什么是 Elixir/OTP

一旦问题变成“怎么做一个长期运行的 agent 系统”，语言和运行时这件事情就绕不过去了。

如果你只是做一个单轮执行器，其实很多技术选型都可以成立。你调一下模型，做一点 tool calling，返回一个结果，这条链路并不要求太多运行时特性。可一旦你想做的东西变成一个长期在线的 agent，事情就不一样了。你会开始关心很多以前不那么显眼的问题：多个聊天通道怎么接进来？不同会话怎么隔离？后台任务怎么跑？出错之后怎么恢复？服务进程之间怎么协作？如果未来 agent 真的会持续升级自己，热更新要不要当成系统能力的一部分？

这些问题放在一起看，`NexAgent` 选择 Elixir/OTP 就不是一个“语言偏好”问题了。

它更像是一种很朴素的匹配关系。OTP 天然就擅长做这种长期活着的系统。监督树让你可以把不同职责拆成服务，把失败当成常态来处理，而不是把整个系统写成一个尽量别崩的大循环；进程隔离让会话、通道、后台任务这些东西有了更自然的边界；并发模型让系统在面对多入口、多任务的时候不会显得特别别扭；热代码加载则给未来的运行时升级留下了空间。

换句话说，`NexAgent` 选择 Elixir/OTP，并不是因为它“适合做 AI”，而是因为它很适合承载一个长期在线、可恢复、可拆分、可热更新的 agent runtime。

## 一个长期 agent 系统，里面到底有什么

如果把 `NexAgent` 往下拆，我更愿意把它理解成几个彼此咬合的层，而不是一张功能清单。

最外面的一层，是入口层。

这一层其实很好理解。`NexAgent` 要接住的，不是一条来自命令行的输入，而是一条持续流动的消息流。消息可能来自 Telegram，可能来自 Feishu，可能来自 Discord、Slack 或 DingTalk。为了让这些入口都能被统一接进系统，`Gateway` 和 channel 适配这一层就必须存在。没有这一层，agent 就只能停留在一个被手动唤起的工具形态；有了这一层，它才开始像一个真的在环境里存在的东西。

再往里一层，是运行层，也就是 `InboundWorker` 和 `Runner`。

从外部看，这两个模块像是在做很朴素的事情：收消息、跑一轮 agent loop。但如果你真的做过，会知道这里面全是细节。消息怎么路由到正确的会话？会话正忙的时候新消息怎么办？后台任务和主会话如何协调？一轮执行到底什么时候该停？工具调用的结果怎么重新回到上下文里？`InboundWorker` 和 `Runner` 的价值，不在于它们“做了什么高级能力”，而在于它们把这些运行时细节组织成了一个稳定的过程。

再往里，就是状态与能力层。

这一层对 `NexAgent` 很关键，因为它决定了这个系统看起来像不像一个真正会积累的 agent。`SessionManager` 负责按 `channel:chat_id` 维护长期会话，这意味着不同聊天入口不会把上下文混在一起。`Memory` 负责长期事实和用户相关信息。`Tool.Registry` 负责工具系统。`Skills` 负责那些可复用的方法和流程。

这里最有意思的，恰恰不是“我们也有 memory，也有 tools”，而是这些东西之间被刻意拉开了边界。很多 agent 系统做到后来，所有新增内容都会被堆成一团：一点 memory、一点 prompt、一点工具、一点 hardcode，时间一长就很难继续长。`NexAgent` 一开始就不太想走这条路，所以它会很强调层次感。

再下一层，是后台执行层。

我一直觉得，很多人谈 agent 的时候，会把“主动性”说得非常抽象，好像非得出现一个能自我规划、自我反思、自我循环的超级智能体，才算主动。但实际做系统时，你会发现主动性往往先从非常具体的结构开始长出来。比如 `Cron`，它给了系统定时执行的能力；比如 `Subagent`，它让系统可以把一个独立问题拆出去，在后台继续做，而不是堵在主会话上。很多原本只能停留在想法里的 agent 行为，到了这一步才开始真正变成系统能力。

最后是进化层。

这一层几乎可以说是 `NexAgent` 最想认真对待的一层。因为如果一个 agent 要长期运行，它迟早会面临同一个问题：新学到的东西，到底该放哪？

## 进化如果不分层，系统很快就会变形

我越来越觉得，agent 系统最怕的一件事，不是能力不够，而是所有变化最后都找不到合适的位置。

今天改一点 prompt，明天记一条 memory，后天又补一个脚本，接着再塞一个工具。短期看系统会越来越“能干”，但只要时间稍微拉长一点，你就会发现这些东西彼此纠缠，难以判断，难以复用，也难以升级。

所以 `NexAgent` 很核心的一个设计，就是把变化明确分层。

`SOUL` 这一层，处理的是 agent 的身份、价值观、长期行为原则。它更像是系统的性格底板。  
`USER` 这一层，处理的是当前用户是谁、他习惯怎样协作、哪些偏好值得长期保留。  
`MEMORY` 这一层，处理的是环境、项目和上下文中的长期事实。  
`SKILL` 这一层，处理的是那些已经稳定下来的方法和流程，也就是“下次遇到类似任务，可以照着做”的程序性知识。  
`TOOL` 这一层，处理的是确定性的可执行能力。当某类动作已经足够稳定，值得从“文本方法”升级成“可执行接口”时，它就应该进入这一层。  
`CODE` 这一层，才处理系统内部实现本身的变化。

这套结构听起来很朴素，但它背后其实有一个很重要的判断：变化应该优先落在更轻、更高的层，而不是一开始就推到底层实现。

能写进 `USER` 或 `MEMORY` 的，就先不要急着做成 `SKILL`。  
能沉淀成 `SKILL` 的，就先不要急着做成 `TOOL`。  
能做成 `TOOL` 的，就先不要急着改 `CODE`。

这套分流的好处，不是“更优雅”这么简单。它真正的价值在于，让一个长期运行的 agent 系统仍然能保持可理解、可迭代。你知道一个变化为什么会出现在这里，也知道它未来应该如何继续演进。

当然，分层不意味着代码层是禁区。

恰恰相反，`NexAgent` 很明确地承认：有些问题最终就是要进入代码层解决。只不过进入代码层，也不意味着系统可以无边界地乱改。它会先通过 `reflect` 去看模块源码、历史和差异，再通过 `upgrade_code` 提交新的实现，然后由 `CodeUpgrade` 负责备份、验证、编译和加载，再由 `UpgradeManager` 去处理热更新、版本保存和回滚保护。

这条路径的意义很直接。系统不回避 code-level evolution，但它希望这种 evolution 是有入口、有过程、有边界的。

## 为什么这些事情在今天值得做

写到这里，问题其实会重新回到最开始。

为什么在 OpenClaw 之后，我们还要再做一个 `NexAgent`？

因为 OpenClaw 让很多人第一次看见了一种可能性，但“看见可能性”和“把它做成一个长期系统”之间，还隔着很长的一段路。那段路里有会话，有记忆，有 skill，有 tool，有后台任务，有权限边界，有 runtime，有升级路径。你不把这些东西一个一个搭起来，agent 很容易就停留在“偶尔看起来很厉害”的阶段。

而我们真正感兴趣的，是再往前走一点点。

我们想知道，agent 能不能从一个好用的 demo，变成一个长期在线的系统。  
我们想知道，agent 能不能真正进入聊天环境，成为日常工作流的一部分。  
我们想知道，agent 能不能不只记住一些标签和片段，而是逐渐学会方法，沉淀方法，复用方法。  
我们想知道，agent 的能力扩展，能不能有一个不会越来越乱的结构。  
我们也想知道，如果这条路继续走下去，未来更有价值的会不会不是一个平均意义上“什么都会一点”的 agent，而是一个能够理解具体用户、服务具体工作流、并在真实环境中持续进化的专属系统。

我觉得，`NexAgent` 至少是在认真回答这些问题。

它不一定已经给出了最终答案，但它确实在试图把这些问题落到一个真实可运行的系统里。

而这件事情本身，就是它存在的意义。
